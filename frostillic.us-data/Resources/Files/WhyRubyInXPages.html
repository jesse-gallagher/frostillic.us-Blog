Why Ruby in XPages?

The most obvious question that comes to mind when hearing about adding Ruby support to XPages is: why bother? After all, EL, JavaScript, and Java are all capable languages for their purposes, and Ruby doesn't allow you to do anything you couldn't, strictly speaking, do before. What it does allow, however, is the ability to work with a more-elegant language that encourages clean use of its advanced features.

I don't claim to be a suitable advocate for Ruby as a language, but there are plenty of resources online. Ruby-doc.org hosts and links to a plethora of information about Ruby is so great, including the Programming Ruby book that I found particularly helpful.

Personally, I'm a fan of the outlook of David Heinemeier Hansson, the creator of Ruby on Rails (a bit on Rails shortly). I've thoroughly enjoyed each presentation he's given, and if you're similar to me, you may enjoy this lecture on the subject of what makes Ruby great.

The best non-specific summary I've found for what makes Ruby different from other languages is this quote from Ruby's creator, Yukihiro Matsumoto:

Ruby is designed to make programmers happy.

First Things First: Does This Mean You Can Use Rails in Domino?

In short: no. Rails is a web application framework rather than a language or an aspect of Ruby itself. Roughly speaking, Ruby is to Rails and Java is to XPages: Rails is implemented in Ruby and is thoroughly influenced by Ruby's aesthetic, but it's a separate software stack of its own.

I like the notion of using Domino as an application server for Rails apps, but that would be, if possible, entirely distinct from XPages.

So What Does This Get You?

What this gets you is a way to write XPages applications using cleaner, more efficient syntax than Java and JavaScript. As an example, you could process a view of documents like so:

regs_view.each_doc do |reg|
	reg.send "recipient@company.com"
	reg["Processed"] = "Yes"
	reg.save
end
